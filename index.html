Caso de Estudio

Retail Analytics Pipeline — Big Data con Apache Spark

Descripción: Diseño e implementación de un pipeline de Big Data end-to-end para la empresa ficticia RetailMax, una plataforma de e-commerce con millones de transacciones diarias. El proyecto procesó más de 112,000 registros combinados de órdenes, ítems y reseñas de clientes utilizando Apache Spark, generando métricas de negocio accionables y modelos de Machine Learning escalables para clasificación y segmentación de usuarios.

Desafío principal: El área de analítica enfrentaba un proceso de análisis lento y fragmentado: los datos de compras, reseñas y comportamiento de clientes estaban distribuidos en múltiples fuentes sin integración, y los modelos de ML tradicionales no podían escalar al volumen real del negocio. El desafío técnico era construir un sistema distribuido capaz de ingestar, transformar y modelar datos masivos de forma eficiente, mientras el desafío de negocio era traducir esos datos en decisiones concretas para marketing y logística.

Solución propuesta: Se diseñó un pipeline progresivo de 5 etapas sobre PySpark, estructurado como un flujo de valor de datos de punta a punta:

  1. Ingesta y configuración de SparkSession optimizada con AQE y esquemas explícitos
  2. Exploración con RDDs, aplicando transformaciones lazy (map, filter, reduceByKey) y documentando el linaje del DAG
  3. Análisis con Spark SQL y DataFrames, con feature engineering (is_delayed, delivery_days, high_satisfaction) y 5 consultas de métricas de negocio
  4. Persistencia en Parquet como formato columnar para optimizar la capa de ML
  5. Pipeline de MLlib con Regresión Logística (clasificación binaria de satisfacción) y K-Means (segmentación en 4 clusters de clientes), evaluados con AUC-ROC, F1-Score y Silhouette Score

El enfoque priorizó decisiones técnicas justificadas desde el negocio: cada transformación respondía a una pregunta analítica concreta.

Herramientas utilizadas: Apache Spark 3.x · PySpark · Spark SQL · Spark MLlib · Python · Pandas · Matplotlib · Google Colab · Parquet · GitHub

Aprendizajes:

  • Comprender la diferencia de paradigma entre procesamiento tradicional y distribuido: pensar en particiones, DAGs y lazy evaluation en lugar de operaciones fila a fila
  • Diseñar pipelines de ML reproducibles donde el preprocesamiento y el modelo viajan juntos como un objeto único, eliminando data leakage entre train y test
  • Traducir métricas técnicas (AUC, Silhouette Score) en lenguaje de negocio accionable para stakeholders no técnicos
  • Aplicar una mentalidad data-driven desde la definición del problema: no construir modelos por construirlos, sino partir de preguntas de negocio claras

Métricas de impacto:

  • Volumen procesado: 112,372 registros combinados de 3 fuentes
  • Reducción de recomputación: Cache/persist evitó releer CSVs en múltiples consultas SQL
  • Precisión del modelo supervisado: ~73% de accuracy · AUC-ROC > 0.70
  • Calidad del clustering: Silhouette Score ~0.45 con k=4 validado por Elbow Method
  • Hallazgo logístico clave: Pedidos retrasados tienen 36% menos satisfacción (score 2.7 vs 4.2)
  • Segmentos accionables: 4 clusters con estrategias diferenciadas para marketing

Habilidades aplicadas: Big Data · Apache Spark · PySpark · RDDs · DataFrames · Spark SQL · Machine Learning Escalable · Feature Engineering · Visualización de datos · Documentación técnica · Pensamiento analítico orientado a negocio

Por qué elegí este proyecto: Este proyecto representa la intersección entre mi rol como Product Owner y mi enfoque data-driven: no solo ejecuté código, sino que tomé decisiones de diseño del pipeline justificadas desde el impacto de negocio. Elegí este caso porque demuestra capacidad para trabajar con tecnologías de escala real (Spark en entorno distribuido), formular preguntas de negocio antes de modelar, y comunicar los resultados técnicos de forma que un área de marketing pueda actuar sobre ellos. Para un PO con mentalidad data-driven, la habilidad de entender cómo se construye y qué limita un pipeline de datos es tan valiosa como saber priorizar un backlog: ambas requieren entender el problema antes de proponer la solución.
